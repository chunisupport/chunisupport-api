# リカバリーコード仕様

## 目的

- メールを使わない運用方針のため、パスワードを失った場合の復旧手段を提供する。
- 個人開発で運用コストを増やさず、かつセキュアに復旧できる仕組みとする。

## 基本方針

- **ワンタイムコード方式**: 1コードは1回のみ使用可能。
- **複数コード発行**: 1ユーザーあたり固定本数（例: 10本）を発行。
- **生コードは保存しない**: サーバーにはハッシュのみ保存。
- **使用後は即時削除**: 使用済みコードはDBから物理削除する。
- **再発行時は旧コードを全無効化**: 既存コードは即時失効。
- **保存促進**: 発行レスポンス以外でコードを再表示しない。

## 仕様（ユーザー体験）

1. 初回登録または設定画面から「リカバリーコード発行」を実行。
2. クライアントはコード一覧を表示し、ユーザーに保存を促す。
3. ログイン不能時は「リカバリーコード + 新しいパスワード」で復旧。

## セキュリティ要件

- コードは十分なエントロピーを持つ乱数で生成する。
- 連続試行を防ぐため、IPベースのレートリミットを必須とする。
- 失敗時は「ユーザー不在/コード不正」の詳細を隠蔽する。
- 監査ログに生コードを残さない。

## データ設計

### テーブル: `user_recovery_codes`

| カラム | 型 | 説明 |
| --- | --- | --- |
| `id` | INT UNSIGNED | 自動採番ID (主キー, AUTO_INCREMENT) |
| `user_id` | INT | ユーザーID (FK, INDEX) |
| `code_hash` | BINARY(32) | コードのSHA-256ハッシュ (UNIQUE INDEX) |
| `created_at` | DATETIME | 作成時刻 |

### 容量に関する配慮

- **1ユーザーあたり最大10件**に固定し、無制限に増えないようにする。
- `code_hash` は **BINARY(32)** を採用してサイズを固定。
- 再発行時は **過去コードを削除**し、保存件数を常に一定に保つ。

## リカバリーコード形式

- 表示用: **12文字の大文字英数字**の `XXXX-XXXX-XXXX` を推奨（例: `A1B2-C3D4-E5F6`）。
- 保存用: 文字列を正規化（ハイフン除去、大文字化）してからハッシュ化。

## バリデーション

- `new_password` は既存のパスワードポリシーに従う。
- `recovery_code` は指定フォーマットに一致しない場合 `bad_request`。

## エラー方針

- 失敗時は以下を同一エラーに統一（ユーザー列挙防止）。
  - `user_not_found`
  - `recovery_code_invalid`
  - `recovery_code_used`
- エラーレスポンスは `invalid_recovery_credentials` に統一。

## 運用ルール

- リカバリーコード発行は **明示操作のみ**で実施。
- 画面側で「紛失時は復旧不可」と明示する。

## 発行フロー設計

1. 認証済みユーザーが `POST /internal/me/recovery-codes` を実行する。
2. サーバーは既存のリカバリーコードを全削除し、新しいコードを10件生成する。
3. レスポンスでコード一覧を返却し、再表示は行わない。
