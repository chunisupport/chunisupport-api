# AGENTS.md 運用ガイドライン

## 基本ルール

### 言語・コミュニケーション

- 必ず日本語で応答してください。
- 指示のない限りコメントは必ず日本語で書いてください。

### 品質保証プロセス

- 作業は必ず `go test` を通した上で、`gofmt` を実行するようにしてください。
- テストを通さないことは許されないので、成功するまでやり直してください。
- 全ての作業が終わった後は、必ず `AGENTS.md` の内容に基づいてセルフレビューを実施し、改善を再度行うというプロセスを3回繰り返してください。各改善プロセスが1回終わるたび、必ず `go test` と `gofmt` を通してください。

### テストケース管理

- テストケースは明確な理由を説明できない・もしくは指示のない限り絶対に変更・削除してはいけません。
- ただし不要なテストケースを増やしてもいけません。同じ問題をテストするケースが重複しないように注意してください。

### コード統一性

- 既存のコードスタイルに従ってください。例えば、変数名や関数名の命名規則、エラーハンドリングの方法などです。
- 既存のコードベースに一貫性を持たせるため、プロジェクト全体で使用されているライブラリやフレームワークを優先的に使用してください。

### パフォーマンス

- パフォーマンスは重要事項です。データベースでN+1問題を起こすコードは絶対に書かないでください。

### ドキュメント管理

- API周りを変更したら必ずその変更を `API.md` などのAPIに関するドキュメントに反映してください。
- APIに関係ないドキュメントの変更も、関連するドキュメントがあれば必ず反映してください。

### 実装原則

- 小さなアプリケーションであるため、複雑さは避け、過剰な実装をしないでください。シンプルな実装を心がけてください。ただしこれは、DDDやクリーンアーキテクチャの原則を犠牲にすることを意味しません。
- 車輪の再発明は避けるようにしてください。ソートアルゴリズムなどを自分で書くことは避け、標準ライブラリや信頼できる外部ライブラリを使用してください。

### 技術スタック

- 本バージョンで利用しているのはGo 1.25系の最新版です。Goの最新バージョンの機能を積極的に活用してください.

## Git運用ルール

### ブランチ名とコミットメッセージのプレフィックス

ブランチ名およびコミットメッセージには、以下のプレフィックスを必ず付与してください。
ブランチ名は `prefix/` (例: `feat/add-new-feature`)、コミットメッセージは `prefix: ` (例: `feat: 新機能を追加`) の形式とします。

- **feat**: 新機能
- **update**: 更新・変更（既存仕様の変更）
- **fix**: 修正（間違いを正したときのみ）
- **docs**: ドキュメントのみ
- **refactor**: リファクタリング（ロジックの変更がないとき）
- **chore**: その他

### ファイル移動

ファイルの移動には `git mv` コマンドを使用してください。

## コーディング規約

### minに関する書き方

- **古い例**
    ```go
    end := start + chunkSize
    if end > len(records) {
        end = len(records)
    }
    ```
- **新しい例**
    ```go
    end := min(start + chunkSize, len(records))
    ```

### interface/anyに関する書き方

- **古い例**
    ```go
    interface{}
    ```
- **新しい例**
    ```go
    any
    ```

### 浮動小数点数の型に関する書き方

**float32の使用は原則禁止です。**

- **理由**: `float32`は精度が低く、例えば`13.4`などの値を正確に表現できません。レーティング計算などの数値演算で誤差が蓄積し、切り捨て処理で期待値と異なる結果になる可能性があります。
- **原則**: すべての浮動小数点数は`float64`を使用してください。
- **例外**: 外部ライブラリのインターフェースで`float32`が要求される場合のみ、その箇所に限定して使用を許可します。ただし、内部処理では必ず`float64`に変換して計算してください。

- **禁止例**
    ```go
    type ChartConstant float32  // 精度の問題が発生
    const := float32(13.4)      // 13.39999961853...
    ```
- **推奨例**
    ```go
    type ChartConstant float64  // 十分な精度を確保
    const := float64(13.4)      // 正確に表現
    ```

### 難易度の文字列表記

**難易度(Difficulty)の文字列は常に大文字で扱います。**

- **理由**: マスターデータのキャッシュキーが大文字で統一されているため、小文字で検索するとリソースが見つからないエラーが発生します。
- **原則**: 難易度を表す文字列(BASIC, ADVANCED, EXPERT, MASTER, ULTIMA)は、データベース内、コード内、API入出力のいずれにおいても**必ず大文字**で扱ってください。
- **禁止事項**: 難易度の検索や比較に `strings.ToLower()` を使用しないでください。必要に応じて `strings.ToUpper()` を使用してください。

- **禁止例**
    ```go
    // 小文字化して検索（エラーになる）
    diffItem, ok := masters.difficulties[strings.ToLower(diffName)]
    ```
- **推奨例**
    ```go
    // 大文字化して検索（正しい）
    diffItem, ok := masters.difficulties[strings.ToUpper(diffName)]
    
    // または元データが既に大文字なら変換不要
    diffItem, ok := masters.difficulties[diffName]  // diffName = "ULTIMA"
    ```

## 設計・実装方針およびリファクタリング指針

本文書は、`chunisupport-api` プロジェクトにおけるコードの「あるべき姿」を定義するものです。
今後の新規開発およびリファクタリングは、本文書に記載された原則に厳格に従って行う必要があります。

### 1. アーキテクチャ原則 (Architectural Principles)

本プロジェクトは **Clean Architecture** を採用します。以下の依存関係のルールを絶対とし、例外を認めません。

#### 依存関係の方向
依存は常に「外側」から「内側」に向かってのみ許可されます。

- **Domain Layer (内側)**: 外部のいかなるレイヤー（Usecase, Infra, App）にも依存してはいけません。
- **Usecase Layer**: Domain Layer にのみ依存します。**Infra Layer (Repositoryの実装やDB構造) に依存することは厳禁です。**
- **Interface Adapters (Infra/Presentation)**: Usecase Layer および Domain Layer に依存します。

#### 禁止事項
- `internal/usecase` パッケージが `internal/infra` パッケージを import すること。
- ドメインロジックがコントローラー（Handler）やインフラ層に流出すること。

### 2. ドメイン駆動設計 (Domain-Driven Design)

ドメインモデルはビジネスロジックの中核であり、最も保護されるべき資産です。

#### ドメインエンティティ (Entity)
- **純粋性**: エンティティは純粋な Go の構造体でなければなりません。`db:"..."` や `json:"..."` といったインフラやプレゼンテーション層の都合によるタグを含めてはいけません。データベース永続化用には `internal/infra/models` パッケージで専用のデータモデルを定義し、`ToEntity()`/`FromEntity()` メソッドで相互変換を行います。
- **豊かなモデル (Rich Model)**: エンティティは単なるデータの入れ物（貧血症モデル）であってはいけません。ビジネスルールやバリデーションロジックをメソッドとして持ち、自身の整合性に責任を持つ必要があります。
  - 悪い例: `Service` 層で `User.IsDeleted` をチェックする。
  - 良い例: `User` エンティティに `IsActive()` メソッドを持たせ、そこで判定する。
- **振る舞いの分類**:
  - **クエリメソッド**: 状態を問い合わせるメソッド（例: `User.IsActive()`, `PlayerRecord.IsRanked()`）
  - **コマンドメソッド**: 状態を変更するメソッド（例: `User.ChangePrivacy()`, `User.Delete()`）。これらのメソッドは内部で `UpdatedAt` を更新するなど、ドメインのルールを強制します。

#### 値オブジェクト (Value Object)
- **活用**: 氏名、ID、パスワードハッシュなど、独自のバリデーションルールや性質を持つ値は、プリミティブ型（string, int）ではなく、専用の構造体（Value Object）として定義してください。
- **不変性**: 値オブジェクトは不変（Immutable）であるべきです。
- **生成**: 必ずコンストラクタ（Factory関数）を通じ、バリデーション済みの状態でのみインスタンス化できるようにしてください。`panic` を起こす `Must` 系の関数は、テストコード以外での使用を極力避けてください。

#### リポジトリ (Repository)
- **集約の永続化**: リポジトリは「テーブルごとのCRUD」ではなく、「集約（Aggregate）の保存と復元」を担当します。部分更新メソッド（`UpdatePrivacy`, `UpdatePassword`など）は廃止し、エンティティのコマンドメソッド + `Save(ctx, entity)` パターンに統一します。
  ```go
  // 推奨パターン
  user, _ := repo.FindByID(ctx, userID)
  user.ChangePrivacy(true)
  repo.Save(ctx, user)
  
  // 非推奨（廃止済み）
  // repo.UpdatePrivacy(ctx, userID, true)
  ```
- **インターフェース**: リポジトリのインターフェースは `internal/domain/repository` で定義し、ドメインエンティティを入出力に用います。全メソッドは第一引数に `context.Context` を取ります。
- **実装の隠蔽**: 実装詳細（SQL, トランザクションオブジェクトの型など）をインターフェースに露出させてはいけません。`Save` メソッドは内部でINSERT/UPDATE判定を行い、適切なSQL文を実行します。

### 3. インフラストラクチャ・実装詳細 (Infrastructure & Implementation)

#### データベース操作
- **明示的なカラム指定**: `SELECT *` の使用は厳禁です。必要なカラムを必ず明示的に指定してください。これはパフォーマンス最適化だけでなく、予期せぬスキーマ変更に対する安全性や、機密情報の意図しない取得を防ぐためです。
- **N+1問題の回避**: ORMの遅延ロード機能に頼らず、適切なJOINやバッチ取得を用いてN+1問題を回避してください（バルク更新APIなど、明示的な例外として認められた箇所を除く）。

#### コンテキスト管理
- **Contextの伝播**: データベース操作を含む全てのI/O処理を行うメソッドは、第一引数に `context.Context` を受け取る必要があります。これにより、リクエストのキャンセルやタイムアウトを適切に伝播させます。

#### エラーハンドリング
- **独自エラー**: ドメイン層で定義された独自エラー（例: `ErrUserNotFound`）を使用し、インフラ層のエラー（例: `sql.ErrNoRows`）をラップまたは変換して返してください。
- **パニック禁止**: ライブラリのバグや起動時の致命的な構成ミスを除き、実行時に `panic` を発生させてはいけません。必ず `error` を返して呼び出し元で処理してください。

### 4. セキュリティ (Security)

- **暗号化**: パスワードや機密情報のハッシュ化には、MD5やSHA1などの脆弱なアルゴリズムを使用しないでください。Argon2id などの安全なアルゴリズムを使用してください。
- **ID生成**: 衝突のリスクがあるハッシュベースのID生成は避け、UUID v4 や CUID、またはDBのシーケンスなど、衝突耐性の高いID体系を採用してください。
- **入力検証**: 外部からの入力は信頼せず、境界（Controller/Interface Adapter）で厳格に検証してください。

### 5. コーディング規約 (Coding Standards)

- **命名**: パッケージ名や構造体名は、その責務を明確に表すものにしてください。`Service` と `Usecase` が混在しないよう、プロジェクト内で用語を統一してください（本プロジェクトでは `Usecase` を推奨）。
- **コメント**: 公開関数や複雑なロジックには、日本語で「なぜそのような実装になっているか」を含めたコメントを記述してください。
- **定数**: 基本的に、値のハードコーディングはせず、info.go内に切り出して利用してください。設定値としてユーザーが変更できるべきと考えるのであれば、それに言及し、例示された設定ファイルを書き換えてください。

### 6. テスト駆動開発（TDD）

原則としてテスト駆動開発（TDD）を採用します。以下のサイクルを厳格に守ってください。

1. **Red**: テストを先に書く。実装前に、期待する振る舞いをテストで定義しましょう。
2. **Green**: テストが成功するための最小限の実装を行う。
3. **Refactor**: テストが成功した状態で、コードの品質を向上させます（既存のテストを通さない変更は厳禁）。

**TDDの利点**:
- テストが実装仕様書として機能し、意図を明確にします。
- 実装と同時にテストが完成するため、テストカバレッジが向上します。
- バグの早期発見につながります。
- リファクタリング時の安心感が増します。

**例外**:
- UI や 統合テストなど、技術的な制約でテストが難しい場合は、その理由を明記してください。
- レガシーコードの緊急修正など、やむを得ない場合は判断してください。ただし、可能な限りテストを補足してください。
