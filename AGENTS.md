# AGENTS.md 運用ガイドライン

## 1. 最重要・基本ルール

### 開発環境とファイルエンコード
- **すべてのファイルは UTF-8 エンコード**で記述・保存してください。
- 特に Windows (PowerShell) 環境でファイルを読み書きする際は、デフォルトのエンコードによる文字化けを防ぐため、必ず明示的に UTF-8 を指定してください。
  - 例: `Get-Content -Path "path/to/file" -Encoding utf8 -Raw`
- コメントや文字列が文字化けしていないか、作業後に必ず確認してください。

### 言語・コミュニケーション
- 応答およびコード内のコメントは、指示のない限り**すべて日本語**で記述してください。

### 実装・設計の基本原則
- 小規模なアプリケーションであるため、過剰な実装を避け、シンプルなコードを心がけてください。
- 車輪の再発明は禁止します。ソートアルゴリズムなどは自作せず、標準ライブラリや信頼できる外部ライブラリを優先してください。
- 明確な理由がない限り、冗長な if 文による不要なチェックは避けてください。

## 2. 設計・アーキテクチャ原則

本プロジェクトは **Clean Architecture** および **ドメイン駆動設計 (DDD)** を採用しています。以下のルールを絶対とし、例外は認めません。

### アーキテクチャの依存規則
依存関係は常に「外側」から「内側」に向かってのみ許可されます。
- **Domain Layer (内側)**: 外部のいかなるレイヤー（Usecase, Infra, App）にも依存してはいけません。
- **Usecase Layer**: Domain Layer にのみ依存します。**Infra Layer (Repositoryの実装やDB構造) への依存は厳禁**です。
- **Interface Adapters (Infra/Presentation)**: Usecase と Domain Layer に依存します。
- **禁止事項**: `internal/usecase` が `internal/infra` を import すること、およびドメインロジックが Controller (Handler) やインフラ層に流出すること。

### ドメインエンティティ (Entity)
- **純粋性の維持**: 純粋な Go の構造体とし、`db:"..."` や `json:"..."` などのインフラ・プレゼンテーション層のタグを含めないでください。DB永続化には `internal/infra/models` の専用モデルを定義し、`ToEntity()` / `FromEntity()` メソッドで相互変換してください。
- **豊かなモデル (Rich Model)**: 単なるデータの入れ物（貧血症モデル）にせず、ビジネスルールやバリデーションロジックをメソッドとして持たせてください（例: `User.IsActive()` など）。

### 値オブジェクト (Value Object)
- 氏名、ID、パスワードハッシュなど、独自のバリデーションルールを持つ値は専用の構造体（Value Object）として定義してください。
- 値オブジェクトは不変（Immutable）とし、必ずコンストラクタ（Factory関数）を通じてバリデーション済みの状態でのみインスタンス化させてください。※テストコードを除き、`panic` を起こす `Must` 系の関数は避けてください。
- **特例**: 値オブジェクトの `Valuer` / `Scanner` インターフェースの実装は、コードのシンプルさと保守性を考慮して許可します。

### リポジトリ (Repository)
- **集約の永続化**: 「テーブルごとのCRUD」ではなく「集約（Aggregate）の保存と復元」を担当します。部分更新メソッド（`UpdatePrivacy` など）は廃止し、エンティティの変更メソッド + `Save(ctx, entity)` パターンに統一してください。
- インターフェースは `internal/domain/repository` で定義し、すべてのメソッドの第一引数に `context.Context` を含めてください。SQLやトランザクションオブジェクトなどの実装詳細をインターフェースに露出させてはいけません。

### データベース・I/O操作
- **明示的なカラム指定**: `SELECT *` の使用は厳禁です。必要なカラムを必ず明示的に指定してください。
- **N+1問題の回避**: パフォーマンスは重要事項です。ORMの遅延ロードに頼らず、適切なJOINやバッチ取得を用いてN+1問題を絶対に回避してください。
- **過去マイグレーションの不変性**: 既に作成済み・適用済みのマイグレーションファイルは、明示的な指示がない限り絶対に変更してはいけません。スキーマ変更が必要な場合は、新しいマイグレーションファイルを追加してください。

## 3. コーディング規約 (Go 1.26+)

本プロジェクトでは **Go 1.26** を利用しています。最新の言語機能を積極的に活用してください。

### 命名・コメント・定数
- **命名**: パッケージ名や構造体名は責務を明確にしてください（本プロジェクトでは `Service` ではなく `Usecase` を使用）。
- **コメント**: 公開関数や複雑なロジックには、日本語で「なぜそのような実装になっているか（意図）」を含めたコメントを記述してください。
- **定数**: 値のハードコーディングは避け、`info.go` 内に切り出して利用してください。

### 組み込み関数・型の使用
- **min / max の記述**: 組み込み関数 `min()` / `max()` を使用してください。
  - *推奨*: `end := min(start + chunkSize, len(records))`
- **any の記述**: `interface{}` は禁止し、`any` を使用してください。

### 浮動小数点数（float32の原則禁止）
- **原則**: すべての浮動小数点数は `float64` を使用してください。`float32` は精度が低く、レーティング計算などで誤差が蓄積する原因になります。
- **例外**: 外部ライブラリのインターフェースで要求される場合のみ限定的に許可します。ただし、内部処理では必ず `float64` に変換して計算してください。

### 難易度の文字列表記（大文字統一）
- 難易度を表す文字列（`BASIC`, `ADVANCED`, `EXPERT`, `MASTER`, `ULTIMA`）は、DB、コード、API入出力のすべてにおいて**必ず大文字**で扱ってください（キャッシュキーが大文字統一されているため）。
- 検索や比較に `strings.ToLower()` を使用することは禁止します。必要に応じて `strings.ToUpper()` を使用してください。

## 4. 品質保証・テスト・レビュープロセス

### 変更時の必須ステップ
1. ソースコードを変更した際は、必ず `go test ./...` を実行し、テストが100%成功することを確認してください。
2. その後、必ず `gofmt -s -w .` を実行してコードを整形してください。ただし、`.git/logs` や `vendor/` ディレクトリなど、プロジェクトのコードベースに含まれないファイルやディレクトリは整形の対象から除外してください。

### セルフレビュー・チェックリスト
作業完了後、提出前に自身のコードに対して以下のチェックリストを用いてセルフレビューを必ず実施してください。
- [ ] `go test` と `gofmt` がエラーなく通っているか？
- [ ] デッドコード、不要なテストケース、不要な引数はすべて削除されているか？
- [ ] コメントや文字列に文字化けはないか？
- [ ] データベース操作で N+1 問題が発生していないか？
- [ ] APIの仕様を変更した場合、`API.md` などの関連ドキュメントに変更を反映したか？

### セルフレビュー（サブエージェント）
サブエージェントが利用可能な場合は、コード変更のセルフレビューをサブエージェントに依頼してください。サブエージェントは、上記のチェックリストを用いてコードをレビューし、問題があれば修正案を提示してください。

### テスト駆動開発 (TDD) とアサーション方針
- 原則として **TDD（Red -> Green -> Refactor）** のサイクルを厳格に守ってください。
- テストケースは、明確な理由や指示がない限り**絶対に変更・削除してはいけません**。また、同じ問題をテストする重複したケースを増やさないでください。
- テストのアサーションは、原則として `github.com/stretchr/testify/assert` を使用してください。
- 後続の検証を継続できない前提条件の確認や `nil` チェック直後のアクセスでは、`github.com/stretchr/testify/require` を使用してください。新規テストでの `t.Errorf` や `t.Fatalf` の直接使用は避けてください。

### テストコードテンプレート（テーブルテスト + Given-When-Then）

```go
func TestPriceCalculator(t *testing.T) {
    tests := []struct {
        name     string
        // Given: 前提条件
        taxRate  float64
        price    int
        // Then: 期待する結果
        expected int
    }{
        {
            name:     "税率10%で1000円の場合1100円になる",
            taxRate:  0.10,
            price:    1000,
            expected: 1100,
        },
        {
            name:     "価格が0の場合は0のまま",
            taxRate:  0.10,
            price:    0,
            expected: 0,
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            // Given
            calc := NewPriceCalculator(tt.taxRate)

            // When
            result := calc.Calculate(tt.price)

            // Then
            assert.Equal(t, tt.expected, result)
        }
    }
}
```

### エラーケースのテストテンプレート

```go
func TestPriceCalculator_Validation(t *testing.T) {
    tests := []struct {
        name    string
        taxRate float64
        price   int
        wantErr bool
    }{
        {
            name:    "負の価格はエラー",
            taxRate: 0.10,
            price:   -100,
            wantErr: true,
        },
        {
            name:    "正常な値はエラーなし",
            taxRate: 0.10,
            price:   1000,
            wantErr: false,
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            calc := NewPriceCalculator(tt.taxRate)
            _, err := calc.CalculateWithValidation(tt.price)

            if tt.wantErr {
                assert.Error(t, err)
            } else {
                assert.NoError(t, err)
            }
        })
    }
}
```

## 5. Git運用ルール

### ブランチ名とコミットメッセージ
ブランチ名には `prefix/`（例: `feat/add-new-feature`）、コミットメッセージには `prefix: `（例: `feat: 新機能を追加`）のプレフィックスを必ず付与してください。

- **feat**: 新機能
- **update**: 更新・変更（既存仕様の変更）
- **fix**: 修正（バグや間違いの修正）
- **docs**: ドキュメントのみの変更
- **refactor**: リファクタリング（ロジックの変更がない修正）
- **chore**: その他雑多なタスク

### ファイル移動
- ファイルを移動・改名する際は、コミット履歴を追跡可能にするため、必ず `git mv` コマンドを使用してください。
