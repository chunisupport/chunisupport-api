# アーキテクチャ概要

このプロジェクトは、クリーンアーキテクチャの原則に部分的に従ったレイヤードアーキテクチャを採用しています。各レイヤーは責務に応じて分割されており、コードの見通しを良くし、保守性とテストの容易性を高めることを目的としています。

依存関係の方向は以下の通りです。

`handler` -> `usecase` -> `repository` (interface) <- `repository` (implementation)

## ディレクトリ構成と各レイヤーの責務

プロジェクトの主要なビジネスロジックは `internal` ディレクトリ内に配置されています。

```
internal/
├── app/         # アプリケーション層: ルーティング、ミドルウェア、DIコンテナ
├── auth/        # 認証・認可に関するドメインロジック
├── config/      # 設定情報の読み込み
├── domain/      # ドメインモデルとリポジトリインターフェース
│   ├── entity/  # エンティティ（純粋なドメインモデル、db/jsonタグなし）
│   ├── vo/      # 値オブジェクト
│   ├── rating/  # レーティング計算ドメインサービス
│   └── repository/ # リポジトリインターフェース
├── dto/         # データ転送オブジェクト（API入出力用）
├── info/        # アプリケーションの基本情報
├── infra/       # インフラストラクチャ層
│   ├── db/      # データベース接続管理
│   ├── models/  # データベース永続化用モデル（db/jsonタグあり）
│   ├── repository/ # リポジトリ実装
│   ├── logger/  # ロギング実装
│   ├── masterdata/ # マスターデータプリロード
│   └── transaction/ # トランザクション管理
├── usecase/     # ユースケース層: アプリケーションのビジネスロジック
└── utils/       # 汎用的なユーティリティ関数
```

### `internal/app` - アプリケーション層

- **役割**: HTTPサーバーの起動、ルーティング、ミドルウェアの設定、依存性注入（DI）など、アプリケーションのセットアップに関連する処理を担当します。
- **具体例**: `echo`フレームワークのインスタンス化、各エンドポイントとハンドラメソッドの紐付け、CORSやロガーといったミドルウェアの適用。

### `internal/domain` - ドメイン層

- **役割**: アプリケーションの中核となるビジネスルールとデータ構造を定義します。このプロジェクトで最も重要な層です。
- **`entity`**: アプリケーションで扱うエンティティ（例: `User`, `Player`）を定義します。**ドメインの純粋性を保つため、`db`タグや`json`タグは一切含まれません**。エンティティは振る舞いメソッド（例: `User.IsActive()`, `PlayerRecord.IsRanked()`）を持ち、ビジネスロジックをカプセル化します。
- **`vo`**: 値オブジェクト（Value Object）を定義します。エンティティの属性を型安全に表現し、不変性とバリデーションを保証します（例: `username.UserName`, `score.Score`）。
- **`repository`**: データベースなどへの永続化処理を抽象化するためのインターフェースを定義します。具体的な実装は`infra/repository`層が担当します。全メソッドは `context.Context` を第一引数に取ります。
- **`rating`**: CHUNITHM特有のレーティング計算ロジックを提供するドメインサービスです（例: `CalcSingleRating`, `CalcSingleOverpower`）。

### `internal/usecase` - ユースケース層

- **役割**: アプリケーション固有のビジネスロジックを実装します。複数のリポジトリを組み合わせて、一つのユースケースを形成します。トランザクション管理やDTO変換など、アプリケーション層の関心事を扱います。
- **具体例**: ユーザー登録処理において、リクエストを受け取り、`UserRepository` を通じてユーザーを保存し、結果を返す、といった一連の流れを実装します。
- **注意**: ドメインロジック自体は `domain` 層に配置し、ここではユースケースの orchestration に集中します。

### `internal/infra` - インフラストラクチャ層

- **役割**: データベース、外部API、ファイルシステムといった、アプリケーションの関心事の外側にある技術的な詳細を実装します。
- **`models`**: データベース永続化専用のデータモデルを定義します。**ドメインエンティティとは別の構造体**であり、`db`タグと`json`タグを持ちます。`ToEntity()`/`FromEntity()`メソッドでドメインエンティティとの相互変換を行います。
- **`repository`**: `domain/repository`で定義されたインターフェースを実装し、具体的なデータ永続化処理（例: SQLの実行）を行います。`sqlx`ライブラリを利用したMySQLへのアクセス処理などがここに記述されます。全メソッドは`GetContext`, `ExecContext`, `QueryContext`などのContext対応メソッドを使用します。
- **`db`**: データベースへの接続に関する処理を担当します。
- **`logger`**: ロギング機能の実装を提供します。
- **`masterdata`**: マスターデータの事前読み込み機能を提供します。
- **`transaction`**: トランザクション管理機能を提供します。

### `internal/app/handler` - ハンドラ層

- **役割**: HTTPリクエストを受け取り、適切なユースケースを呼び出し、レスポンスを生成する責務を持ちます。
- **具体例**: リクエストボディのJSONをパースしてGoの構造体にマッピングし、`usecase`層のメソッドに渡した後、結果をJSON形式でクライアントに返却します。

### `internal/dto` - データ転送オブジェクト

- **役割**: レイヤー間でデータをやり取りするために使用される構造体を定義します。特に、リクエストの入力やレスポンスの出力形式を定義する際に利用します。これにより、ドメインモデルがAPIのインターフェースに直接公開されるのを防ぎます。

### `internal/config`

- **役割**: `.config/<environment>.settings.json` からアプリケーション設定を読み込み、構造体にマッピングする責務を持ちます。

### `internal/auth`

- **役割**: JWT（JSON Web Token）の生成や検証といった、認証・認可に特化したロジックを提供します。

### `internal/info`

- **役割**: アプリケーション名やバージョン、各種定数など、アプリケーション全体で利用される静的な情報を提供します。

### `internal/utils`

- **役割**: 特定のドメインに依存しない、汎用的な補助関数（ヘルパー関数）を提供します。

## ドメイン駆動設計 (DDD) の採用

2025年12月18日のリファクタリングにより、本プロジェクトはドメイン駆動設計の原則を強化しました。

### ドメインの純粋性

- **エンティティの独立性**: `domain/entity`パッケージの構造体は、インフラストラクチャ層の関心事（DB永続化、JSON変換）から完全に分離されています。`db`タグや`json`タグを一切持たず、純粋なドメインモデルとして機能します。

- **Infra層でのマッピング**: データベース永続化のために、`internal/infra/models`パッケージで専用のデータモデルを定義しています。これらのモデルは`ToEntity()`および`FromEntity()`メソッドでドメインエンティティとの相互変換を行います。

### リッチドメインモデル

エンティティは単なるデータ構造ではなく、ビジネスロジックをカプセル化する振る舞いを持ちます：

**クエリメソッド（状態の問い合わせ）**
- `User.IsActive()`: ユーザーが有効（削除されていない）かを判定
- `User.IsPublic()`: ユーザーが公開設定かを判定
- `User.HasLinkedPlayer()`: プレイヤーが紐づいているかを判定
- `PlayerRecord.IsRanked()`: レコードがランキング対象かを判定
- `PlayerRecord.SlotKey()`: スロット種別キーを取得

**コマンドメソッド（状態の変更）**
- `User.ChangePrivacy(isPrivate)`: プライバシー設定を変更
- `User.ChangePassword(hash)`: パスワードを変更
- `User.Delete()`: 論理削除を実行
- `User.LinkPlayer(playerID)`: プレイヤーを紐付け

これらのメソッドは、状態変更時に自動的に`UpdatedAt`を更新するなど、ドメインのルールを強制します。

### 集約指向の永続化

リポジトリパターンは、集約単位での操作を基本とします：

- **Save(ctx, entity)**: 集約ルートを保存します。内部でINSERT/UPDATE判定を行い、適切なSQL文を実行します。
- **部分更新の廃止**: `UpdatePrivacy`や`UpdatePassword`といった特定カラムのみを更新するメソッドは廃止され、エンティティのコマンドメソッド + `Save`パターンに統一されました。

```go
// 推奨されるパターン
user, _ := repo.FindByID(ctx, userID)
user.ChangePrivacy(true)
repo.Save(ctx, user)

// 非推奨（廃止済み）
// repo.UpdatePrivacy(ctx, userID, true)
```

### Context対応

すべてのリポジトリメソッドは`context.Context`を第一引数に取り、タイムアウトやキャンセルをサポートします。DB操作には`GetContext`、`ExecContext`、`QueryContext`などのContext対応メソッドを使用しています。

### ドメインサービス

複数エンティティにまたがる、またはエンティティに属さないドメインロジックは、ドメインサービスとして実装されています：

- **domain/rating**: CHUNITHMのレーティング計算ロジック（`CalcSingleRating`, `CalcSingleOverpower`）
