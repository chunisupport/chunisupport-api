# クリーンアーキテクチャ簡略化候補レポート（2026-05-13）

## 目的

本レポートは、現行コードベースにおいてクリーンアーキテクチャ／DDDの原則を維持しつつ、運用コストを下げるために「簡略化したほうがよい箇所」を整理するものです。

## 前提

- 依存関係の方向（外側→内側）は維持する。
- ドメイン層の純粋性は維持する。
- ハンドラ層へのビジネスロジック流出は行わない。
- 目的は原則の撤廃ではなく、過剰な抽象化の解消である。

## 現状の観察

### 1. Usecaseの interface + 実装分離が一律で適用されている

多くのユースケースで `XxxUsecase` interface と `xxxUsecaseImpl` の構成が採用されています。単一実装しか存在しない期間が長い場合、以下のコストが発生します。

- メソッド追加時の修正点増加（interface / impl / テストダブル）
- リネーム時の追従範囲拡大
- 「差し替えが必要になるまで」発生しない抽象化コストの先払い

#### 簡略化提案

- **提案A（推奨）**: 外部境界として必要な interface のみ維持し、内部利用のみのユースケースは具象型公開へ寄せる。
- **提案B**: interface は残すが、増減が頻繁なユースケースは用途別に分割し、巨大interface化を避ける。

#### 期待効果

- 変更時の編集箇所が減る
- 実装理解が早くなる
- テストダブルの過剰メンテナンスが減る

---

### 2. DTO／入力モデルの多層化により変換コストが高い

API DTO、usecase入力、entity組み立てが複数段存在し、フィールド追加時の追従点が増えています。

#### 簡略化提案

- **提案A（推奨）**: API境界で必要なDTOは維持し、usecase内専用入力モデルは「再利用実益があるもの」に限定する。
- **提案B**: 変換ロジックを機能単位のMapperへ集約し、重複変換を一本化する。
- **提案C**: 変換対象の追加ルールを文書化し、安易な型追加を防ぐ。

#### 期待効果

- フィールド追加時の漏れ防止
- 仕様変更時の修正時間短縮
- 変換起因バグの減少

---

### 3. Router（DI組み立て）の集中化

`NewRouter` が Repository / Usecase / Handler 生成を広く担っており、機能追加時に配線変更が一点へ集中しています。

#### 簡略化提案

- **提案A（推奨）**: 機能単位の Provider 関数に分割する（例: SongModule, UserModule）。
- **提案B**: 依存セット構造体（例: `AppDependencies`）を導入し、コンストラクタ引数の肥大化を抑える。
- **提案C**: ルーティング登録と依存生成をファイル分離して見通しを改善する。

#### 期待効果

- 配線変更の局所化
- コンフリクト減少
- 新規参加者の理解コスト低下

---

## 優先度

1. **高**: Router配線の分割（変更頻度が高く効果が出やすい）
2. **中**: DTO/入力モデル整理（保守コストに直結）
3. **中**: Usecase interface適用範囲の見直し（中長期効果）

## 非推奨（やらない方がよいこと）

- ドメイン層へ `json/db` タグを戻す
- ハンドラへバリデーションやユースケース判断ロジックを戻す
- 一気に全ユースケースの抽象化を剥がす

## 実施方針

- 既存テストが守る振る舞いを維持しながら、小さな単位で簡略化を進める。
- 影響範囲が大きい変更は、先に配線分割で見通しを作ってから行う。
- 変更後は「追加した抽象化が将来の変更コストを下げるか」をレビュー観点に含める。

## 成果確認の観点（KPI例）

- 機能追加時に必要な修正ファイル数
- コンストラクタ引数数の平均
- 1ユースケースあたりの変換関数数
- モック修正件数（interface変更時）
