# `cmb_lv` 定義差分 調査指示書・手順書

## 1. 目的

`cmb_lv`（コンボランプID）の定義について、以下の差分を特定し、**どこが正しいソース・オブ・トゥルースか**を確定する。

- APIドキュメント記載値（例: `0=NONE, 1=FULL COMBO, 2=ALL JUSTICE`）
- ドメインサービス実装値（例: `2=FULL COMBO, 3=ALL JUSTICE`）
- 実データ（DB・マスターデータ・既存APIレスポンス）の実値

あわせて、影響範囲（今回PR内のみか、既存全体か）を切り分ける。

---

## 2. 期待成果物

1. 調査レポート（本ファイル末尾テンプレートを利用）
2. 正式な `cmb_lv` 定義（暫定/確定）
3. 修正方針の提案（最小修正 / 恒久対応）
4. 影響範囲一覧（API・DB・フロント・バッチ・テスト・ドキュメント）

---

## 3. 調査観点（必須）

### 3.1 定義の出どころ

- ドメイン層で `cmb_lv` 相当値を参照している定数・ロジック
- DTO/ハンドラ/ユースケース経由でレスポンス化される値
- `docs/API.md` のスキーマ定義
- DB上の保存値（過去データを含む）

### 3.2 互換性

- 既存クライアントが前提としている値
- 変更時に破壊的変更が発生するか
- バージョン（`/internal` と `/v1`）間の不整合有無

### 3.3 再現性

- 同一入力で、計算ロジックとAPI出力が一致するか
- テストコードの期待値と本番データが一致するか

---

## 4. 手順（実施順）

### Step 1: コード上の定義を機械的に列挙

以下コマンドで `cmb_lv` / combo lamp 関連を横断検索する。

```bash
rg -n "cmb_lv|comboLamp|ALL JUSTICE|FULL COMBO|theoreticalScore|CalcSingleOverpower" internal docs
```

確認ポイント:
- 定数値（数値）
- 比較条件（`==` / `>=`）
- API返却時に値変換（マッピング）している箇所の有無

### Step 2: APIドキュメント定義の確認

`docs/API.md` の `scores.full` など `cmb_lv` 定義箇所を確認し、数値定義を抽出する。

確認ポイント:
- 定義が一箇所か、複数箇所で重複していないか
- `/internal` と `/v1` で差異がないか

### Step 3: DB実データの実態確認

`scores`（または `cmb_lv` を保持するテーブル）で、実際に保存されている値の分布を確認する。

```sql
SELECT cmb_lv, COUNT(*)
FROM scores
GROUP BY cmb_lv
ORDER BY cmb_lv;
```

追加確認（推奨）:
- 直近データのみ / 全期間データ
- 曲別・難易度別での偏り
- 想定外値（0,1,2,3 以外）の有無

### Step 4: エンドポイント実測

`/internal/songs`, `/v1/songs` および `scores` を返す関連APIを実行し、`cmb_lv` 実値を確認する。

確認ポイント:
- ドキュメント通りの値か
- 計算ロジック（`maxop` 等）と整合するか
- 同一ユーザー/同一譜面で値が揺れないか

### Step 5: テスト観点の照合

`cmb_lv` やOP計算に関する既存テストを確認し、期待値と実装・DBの整合を確認する。

```bash
rg -n "cmb_lv|comboLamp|CalcSongMaxOP|CalcSingleOverpower" internal/**/*_test.go
```

確認ポイント:
- 期待値が「仕様」を表しているか、「現実装の追認」になっていないか
- 影響範囲外のテストまで暗黙依存していないか

### Step 6: 原因分類と方針決定

以下のどれかに分類する。

- A: ドキュメント誤り（実装・データが正）
- B: 実装誤り（ドキュメント・データが正）
- C: 層ごとのID定義が異なる（変換層不足）
- D: 履歴的負債（時期により混在）

分類後、最小修正と恒久対応を定義する。

---

## 5. 判断基準（推奨）

- **第一優先**: 実データ互換性（既存ユーザー影響最小）
- **第二優先**: API契約の明確性（ドキュメントと実装一致）
- **第三優先**: アーキテクチャ整合（ドメイン定義と変換層責務）

---

## 6. 対応案テンプレート

### 案1: ドキュメント修正のみ

適用条件:
- DB実値・実装・既存クライアント期待が一致している
- 不整合が `docs/API.md` のみ

実施内容:
- `docs/API.md` の `cmb_lv` 定義を実装値に合わせる
- 変更履歴に「過去記載誤り修正」を明記

### 案2: 変換層導入

適用条件:
- ドメインIDとAPI契約IDを分離したい
- 既存契約維持が必要

実施内容:
- DTO変換時に明示的マッピング関数を導入
- 双方向変換（入力/出力）をテストで固定

### 案3: 実装統一（破壊的変更注意）

適用条件:
- ドキュメント定義を正とする意思決定がある
- クライアント・DB移行計画が確保されている

実施内容:
- 定数・計算ロジック・永続値・テストを一括更新
- 互換期間中は旧値受け入れまたはバージョニング
