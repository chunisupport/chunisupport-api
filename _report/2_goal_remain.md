# Goal機能 残対応メモ（未完了項目のみ）

最終更新: 2026-02-20

## 方針

本メモは、`_report/goal_remain.md` にあった項目のうち、現行実装で未完了のものだけを統合した版である。
対応済みのHigh項目（動的上限検証、`overpower_percent` の割合仕様、`goals.user_id` 型整合、マスタ欠損時の空文字返却防止）は本書から除外している。

## 未完了項目

### 1. `title` バリデーションの境界定義明確化（優先度: Middle）

- 制御文字の拒否は実装済みだが、ゼロ幅文字やUnicode正規化（NFC/NFKC）の扱いは未定義。
- UI表示崩れ・検索性低下・監査ログ可読性低下の予防には、文字種ルールの追加明文化が必要。

**対応方針**
- 許容/禁止文字種を仕様化し、APIドキュメントへ追記する。
- 正規化方針（NFC/NFKCの採否）を決定し、Usecase実装とテストへ反映する。

### 2. `attributes` のキー許容方針に対する互換戦略（優先度: Middle）

- 現状は未知キーをエラーにする厳格運用。
- 将来拡張時に古いクライアントとの互換性に影響しやすいため、運用戦略の明文化が必要。

**対応方針**
- 厳格運用継続か、互換期間中のみ未知キー無視とするかを決定する。
- 採用方針を `docs/API.md` に明記する。

### 3. エラー分類/理由コード運用の明確化（優先度: Middle）

- Goal APIの公開エラーコードは整理済みだが、内部ログの理由コード（`reason`）との対応管理が弱い。
- フロント実装・運用調査の速度向上のため、対応表と更新ルールの明文化が望ましい。

**対応方針**
- エラーコードと内部理由コードの対応表を `docs/` または `_report/` に作成する。
- 変更時に対応表を更新する運用ルールを定義する。

### 4. Goalエンティティの型安全化（優先度: Low）

- 現状は `achievement_params` / `attributes` を `[]byte` / `map[string]any` 中心で扱っている。
- 設計方針（アプリ内部で型安全な構造体として扱う）に対して段階移行が未完了。

**対応方針（段階移行）**
1. Usecase内の `achievement_type` ごとのパラメータを型付きDTOへ統一。
2. Entity/Repository境界へ型付き表現を導入し、JSONとの相互変換を集約。
3. API互換を維持しながら `map[string]any` 依存を順次削減。

### 5. 監視メトリクス不足（優先度: Low）

- バリデーション失敗理由はログに出るが、失敗件数の継続監視ができない。
- achievement_type単位での失敗分布を可視化できず、閾値/仕様の改善サイクルが回しづらい。

**対応方針**
- `achievement_type` / `reason` 単位でカウンタメトリクスを追加する。
- リリース後の失敗分布をもとに、入力仕様・エラーメッセージを継続改善する。
