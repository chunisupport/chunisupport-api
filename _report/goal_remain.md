# Goal機能 残対応・リスク調査メモ

最終更新: 2026-02-20

## 方針更新（合意済み）

- `overpower_percent` は名前どおり「割合（%）」を保持する。
- サーバ内部でも `achievement_params.total` を割合として保存・判定する（0以上100以下・小数3桁まで）。
- `overpower_value` は従来どおり OverPower の実数値（理論値系と同じスケール）を保持する。
- 役割を分けることで、割合指標（`overpower_percent`）と実数値指標（`overpower_value`）の両方を提供する。

## 優先度High（仕様・実装乖離 / エラー原因になりやすい項目）

### 1. サーバサイドの動的バリデーション不足

現状は `achievement_type` ごとの JSON 形状・最低限の範囲チェックが中心で、
`attributes` で絞り込まれた対象譜面集合に基づく**上限値検証**が不足している。

- 例: `score_count.count`, `rank_count.count`, `hardlamp_count.count`, `combolamp_count.count`
  - 「対象譜面数を超える count」を保存できる可能性がある。
- 例: `total_score.total`
  - 「対象譜面数 × 1010000」を超える値を保存できる可能性がある。
- 例: `overpower_value.total` / `overpower_percent.total`
  - 対象譜面の理論値合計を超える値が保存できる可能性がある。

**対応方針**
- Usecase層で `attributes` を解釈し対象譜面集合を確定。
- achievement_type ごとに「入力値 <= 動的上限」を検証。
- 上限算出ロジックはテスタブルな関数に分離し、境界値テスト（=上限, 上限+1）を追加。

### 2. `overpower_percent` の仕様逸脱（割合保持からのずれ）

`overpower_percent` を実数値として扱う方針は、元仕様（割合を保存）と乖離している。
API利用者が「%」として扱う前提を崩すため、表示・判定の双方で不整合が起きやすい。

**対応方針**
- ドメイン仕様を「`overpower_percent` は割合（%）を保持する」と明記。
- API.md / 設計書 / DTO説明を割合仕様へ統一。
- バリデーションを `0 <= total <= 100` に統一し、境界値テストを維持・追加する。

### 3. migrationの型不一致リスク（`goals.user_id`）

調査メモ上は `INT UNSIGNED` に揃える方針だが、DDLで `INT` になっていると
FK作成エラーや将来移行時の不整合を招く。

**対応方針**
- まだ利用者がおらず既存データもないため、最新のマイグレーションファイルを書き換えて `users.id` と同じ符号属性へ統一。
- 変更時はロールバック手順（down migration）も合わせて整備。

### 4. Usecase出力時のマスタ欠損耐性不足

`achievement_type_id` がキャッシュに存在しない場合、警告ログのみで処理継続している。
レスポンス `achievement_type` が空文字になりうるため、クライアント側エラー要因になる。

**対応方針**
- 欠損時は安全側でエラー返却（`internal_error`）するか、`UNKNOWN` を返す仕様に統一。
- DB整合性（FK）とキャッシュ整合性（起動時ロード失敗検知）の二重防御を実施。

## 優先度Middle（不具合予防・運用性向上）

### 5. `title` バリデーションの境界定義をより明確化

- 現状は trim 後 30文字制限だが、改行・制御文字・ゼロ幅文字の扱いが未定義。
- UI表示崩れや検索性低下、監査ログ可読性低下につながる可能性がある。

**対応方針**
- 許容文字種を仕様化（最低限: 制御文字禁止）。
- 正規化方針（NFC/NFKCの採否）を決める。

### 6. `attributes.const` の精度ルール明確化

- `float64` 前提は明示済みだが、入力小数桁（1桁固定か、任意桁許容か）の定義が曖昧。
- 比較誤差により境界譜面の含む/含まない判定が揺れる恐れがある。

**対応方針**
- 桁数制約（例: 小数1桁まで）をサーバ側で明示バリデーション。
- 比較時は丸め基準を統一し、テストに境界値を追加。

### 7. `attributes` のキー許容方針に対する互換戦略不足

現状は未知キーをエラーにする厳格運用。将来拡張時に古いクライアントとの互換を崩しやすい。

**対応方針**
- 厳格運用を維持するか、互換期間中のみ未知キー無視にするか方針決定。
- どちらを採る場合も、API.md に明記する。

### 8. エラー分類の詳細不足（同一コードへ集約しすぎ）

`invalid_goal_input` に多くの失敗要因を集約すると、フロントのユーザー向けエラー表示が粗くなる。

**対応方針**
- 内部ログに理由コードを必ず出す。
- 必要に応じて将来 `invalid_goal_title` / `invalid_goal_attributes` 等へ細分化を検討。

## 優先度Low（将来改善）

### 9. Goalエンティティの型安全化（JSON生バイト脱却）

設計方針上は型安全構造体化が望ましいが、現状は `[]byte` / `map[string]any` 中心。
大規模変更になりやすいため段階的移行が必要。

**対応方針**
- まず Usecase内デコードを型付き構造体に置換。
- 次に Entity/Repository 境界を段階移行。

### 10. 監視メトリクス不足

目標作成・更新失敗の原因分布が可視化されていない。

**対応方針**
- achievement_type ごとのバリデーション失敗件数をログ/メトリクス化。
- リリース後に実利用データで閾値・仕様妥当性を再評価。

---

## エラーが出る可能性のある箇所（重点探索ポイント）

- マスタキャッシュ未初期化・欠損時（起動時ロード失敗、運用データ不整合）。
- JSONデコード失敗（`achievement_params` / `attributes` の型不一致、数値型の過大値）。
- DB制約違反（achievement_type外部キー不整合、将来の型不一致migration）。
- 100件上限付近の並列作成（ロック漏れやトランザクション境界破綻がある場合）。
- 小数比較の境界（`const`, OP実数値）での誤差起因の判定ずれ。

## 実装確認時のチェックリスト（次回レビュー用）

- [ ] `achievement_type` ごとに「構造検証 + 動的上限検証 + 境界値テスト」が揃っている。
- [ ] `overpower_percent` の内部値定義（割合）とAPIドキュメント記載が一致している。
- [ ] `goals.user_id` の符号属性が `users.id` と一致し、migrationで再現できる。
- [ ] マスタ欠損時に空文字を返さず、規定エラーで失敗する。
- [ ] エラーコードと内部ログ理由の対応表が保守されている。

## 次の具体アクション（推奨順）

1. `overpower_percent` を元仕様（割合保存）へ戻し、API.md/設計書/実装の記載を同期。
2. Usecase に動的上限バリデーションを追加し、テストで境界値を固定。
3. `goals.user_id` 型整合のmigrationを修正。
4. マスタ欠損時のエラー戦略（空文字返却を禁止）を統一。
5. タイトル文字種・小数桁数の仕様を明文化してバリデーション追加。
