# OVER POWER集計API 計画書

## 概要

ユーザーの OVER POWER をバックエンドで集計し、API から返却する仕組みを追加する。
対象は以下の 25 項目とする。

- 全曲: 1
- ジャンル別: 7
- 難易度別: 5
- レベル別: 12

合計 25 項目。

また、集計単位は以下の 2 系統に分かれる。

- 全曲・ジャンル別: 楽曲単位集計
- 難易度別・レベル別: 譜面単位集計

## 背景

現状はフロントエンド側で OP 計算を前提としたデータ利用箇所が存在し、`songs.maxop` などの値も提供されている。
一方で、今回必要な集計は単純な譜面 OP 合算ではなく、集計軸によって採用ルールが異なる。

- 全曲・ジャンル別は、同一楽曲の中で「その時点で獲得している単譜面 OP が最大の譜面 1 件のみ」を採用する
- 難易度別・レベル別は、譜面が一意に定まるため譜面 OP をそのまま利用する

このルール差をフロントエンドだけで持つと、実装重複とロジック乖離のリスクが高い。
そのため、集計結果の正本はバックエンド API に置く方針が妥当と考える。

## 方針

### 正本の責務

OVER POWER 集計の正本はバックエンド API とする。

フロントエンドは以下の責務に留める。

- API レスポンスの表示
- 必要であれば即時プレビュー表示

ただし、最終的な共有値・保存値・外部公開値として扱うのは API レスポンスとする。

### 初回リリース範囲

初版は本人向け API のみ提供する。

候補:
- `POST /internal/me/overpower-summary`

公開プロフィール向け API は後続対応とする。
理由は以下。

- まずは UI 検証用の本人向け API があれば十分価値がある
- 公開 API 化すると、公開/非公開ルールやレスポンス互換性の考慮が増える

## 集計ルール

### 全曲・ジャンル別

集計単位は楽曲単位とする。

各楽曲について、その楽曲に紐づく譜面のうち、ユーザーが現在保持している単譜面 OP が最大の 1 譜面だけを採用する。

つまり、同一楽曲内で複数譜面の OP を合算しない。

- current OP:
  - その曲で現在獲得している単譜面 OP の最大値
- max OP:
  - その曲の理論最大 OP

### 難易度別・レベル別

集計単位は譜面単位とする。

譜面は難易度またはレベル帯によって一意に分類できるため、単譜面 OP をそのまま集計に使う。

- current OP:
  - 対象譜面の現在 OP の合計
- max OP:
  - 対象譜面の理論最大 OP の合計

## レベル別の判定基準

レベル別は譜面定数 `const` から一意に決定する。

判定方法:
- `floor(const * 2) / 2` で 0.5 刻みへ切り下げる

例:
- `14.9` -> `14+`
- `14.5` -> `14+`
- `14.4` -> `14`
- `15.7` -> `15+`

初版では 10〜15+ のみを集計対象とする。

## 未解禁曲の扱い

未解禁曲の管理は OVER POWER 集計 API の責務に含めない。

将来的に未解禁曲登録取得 API を別途用意し、ユーザーごとの解禁状態はその系統で扱う前提とする。
そのため本 API は、現時点では未解禁曲を意識しない単純な集計 API として設計する。

この方針により、OP 取得処理は楽曲マスタとプレイヤーレコードだけを入力として完結させる。
未解禁曲を表示上どう扱うか、あるいはどの画面で管理するかは、別 API とフロントエンド側の責務として切り分ける。

## レスポンス方針

初版は、表示に必要な集計値を過不足なく返すシンプルなレスポンスとする。

### 理由

単なる current OP のみでは、フロントエンドが表示に必要な割合や残量を再計算する必要がある。
一方で、項目ごとに固定で決まる情報や初版で使わない補助情報まで返す必要はない。

そのため、以下を返す方針とする。

- current OP
- max OP
- percent
- 対象数
- プレイ済み数

## 実装方針

### Usecase

OP 集計専用 Usecase を新設する。

責務:
- プレイヤーレコード取得
- 通常楽曲マスタ取得
- 25 項目集計
- レスポンス DTO 組み立て

既存の `UserUsecase` に混ぜず、専用 Usecase として切り出す。

### Repository 利用

既存の以下を利用する想定。

- `PlayerRecordRepository.FindByPlayerID`
- `SongRepository.FindAllExcludingWorldsend`

初版は追加クエリなしでも成立する見込み。

### 集計アルゴリズム

1. プレイヤーの通常譜面レコードを取得
2. 全通常楽曲と譜面マスタを取得
3. 楽曲ごとの現在最大単譜面 OP を算出
4. 楽曲ごとの理論最大 OP を算出
5. 全曲・ジャンル別へ楽曲単位で集計
6. 難易度別・レベル別へ譜面単位で集計
7. DTO に変換して返却

## 負荷見積もり

初版の負荷は問題ない見込み。

理由:
- 対象は通常譜面全件でも数千件規模
- DB アクセスは実質 2 系統
  - プレイヤーレコード取得
  - 楽曲マスタ取得
- 集計処理は Go 側の map ベース集約で完結する

高頻度連打されない限り、初版はキャッシュなしでも成立可能と考える。

必要になれば以下を後続で検討する。
- 楽曲マスタ側のキャッシュ活用
- レスポンスキャッシュ

## メリット

- 集計ルールを backend に一元化できる
- フロントとのロジック乖離を防げる
- 将来、他クライアントや公開 API へ再利用しやすい
- OVER POWER 集計と未解禁曲管理の責務を分離できる
- 返却項目を最小限に絞り、API を単純に保てる

## 懸念点

- フロントだけで完結するよりはレスポンス待ちが発生する
- 未解禁曲を考慮した表示や管理は別 API 側の設計が必要になる
- レベル別が const ベース前提であるため、将来ゲーム仕様とズレる場合は修正が必要

ただし、これらは初版の価値を損なうほどの問題ではない。

## 今回の結論

- OVER POWER 集計 API を作る意義はある
- 正本はバックエンドに寄せるのが妥当
- 初版は本人向け API のみ
- 未解禁曲は OVER POWER 集計 API では扱わない
- レベル別は const から 0.5 刻み切り下げで判定
- レスポンスは current/max/percent と件数情報に絞る
